## **1. Веб-серверы и прокси**
**Назначение:**
- Обработка HTTP-запросов и обслуживание клиентов
- Маршрутизация трафика между сервисами
- Терминация SSL/TLS соединений

**Популярные решения:**
- **Nginx**: Высокопроизводительный веб-сервер и reverse-proxy
- **Apache**: Модульная архитектура, широкие возможности кастомизации
- **Caddy**: Автоматический HTTPS, простота конфигурации
- **Traefik**: Облачный reverse-proxy с интеграцией с Docker/Kubernetes
- **Envoy**: Высокопроизводительный прокси для сервисных сеток

**Ключевые возможности:**
- Раздача статических файлов
- Виртуальные хосты и маршрутизация
- Балансировка нагрузки
- Кэширование ответов
- Rate limiting и защита от DDoS

Подробнее: [[Веб-серверы и прокси]]

## **2. Балансировщики нагрузки**
**Назначение:**
- Распределение трафика между инстансами приложения
- Обеспечение отказоустойчивости и масштабирования
- Обнаружение неработающих серверов (health checks)

**Типы и решения:**
- **Аппаратные**: F5, Citrix ADC (NetScaler)
- **Программные**: HAProxy, Nginx, Pound
- **Облачные**: AWS ALB/ELB/NLB, Google Cloud Load Balancing, Azure Load Balancer
- **Kubernetes Ingress**: Ingress Controllers (NGINX Ingress, HAProxy Ingress)
- **Сервисные сетки**: Istio, Linkerd (балансировка на стороне клиента)

**Алгоритмы распределения:**
- Round Robin, Least Connections, IP Hash
- Взвешенные алгоритмы
- Географическая маршрутизация

Подробнее: [[Балансировщики нагрузки]]

## **3. Системы кэширования**
**Назначение:**
- Хранение часто запрашиваемых данных для быстрого доступа
- Уменьшение нагрузки на базы данных и бэкенды
- Снижение задержек ответа

**Типы систем кэширования:**
- **In-memory хранилища**: Redis, Memcached, Dragonfly
- **HTTP-кэши**: Varnish, Squid, Fastly
- **CDN**: CloudFlare, Akamai, AWS CloudFront
- **Database-встроенные**: Кэши запросов PostgreSQL, MySQL

**Сценарии использования:**
- Кэширование результатов запросов к БД
- Хранение сессий пользователей
- Кэширование API-ответов
- Геораспределенная доставка контента

**Стратегии инвалидации:**
- TTL (Time To Live)
- Write-through / Write-behind
- Cache-aside (ленивая загрузка)

Подробнее: [[Системы кэширования]]

## **4. Брокеры сообщений**
**Назначение:**
- Асинхронная обработка задач и событий
- Слабая связанность (decoupling) компонентов системы
- Буферизация пиковых нагрузок

**Популярные решения:**
- **Kafka**: Распределенная платформа для стриминга событий
- **RabbitMQ**: Брокер с поддержкой AMQP, гибкой маршрутизацией
- **NATS**: Легковесный, высокопроизводительный брокер
- **ActiveMQ**: Классический брокер с поддержкой JMS
- **AWS SQS/SNS**: Управляемые очереди и нотификации в облаке

**Модели взаимодействия:**
- Очереди (point-to-point)
- Публикация/подписка (pub/sub)
- Маршрутизация по темам (topics) и ключам (routing keys)
- Потоковая обработка (stream processing)

**Сценарии использования:**
- Обработка фоновых задач (email, нотификации)
- Сбор и агрегация логов
- CQRS и event sourcing
- Микросервисная коммуникация

Подробнее: [[Брокеры сообщений]]

## **5. Контейнеризация и оркестрация**
**Назначение:**
- Изоляция приложений и их зависимостей
- Упрощение деплоя и масштабирования
- Управление жизненным циклом приложений

**Контейнеризация:**
- **Docker**: Стандарт де-факто для контейнеризации
- **Podman**: Бездемонная альтернатива Docker
- **containerd**: Промышленный runtime (используется в Docker)
- **CRI-O**: Реализация CRI для Kubernetes

**Оркестрация:**
- **Kubernetes**: Промышленный стандарт оркестрации
- **Docker Swarm**: Встроенная оркестрация в Docker
- **Nomad**: Простая оркестрация от HashiCorp
- **Amazon ECS / Google Cloud Run**: Управляемые сервисы

**Экосистема:**
- **Docker Compose**: Локальная оркестрация для разработки
- **Helm**: Пакетный менеджер для Kubernetes
- **Service Mesh**: Istio, Linkerd для управления трафиком

Подробнее: [[Контейнеризация и оркестрация]]

## **6. Базы данных в инфраструктуре**
**Назначение:**
- Персистентное хранение данных
- Обеспечение консистентности и доступности

**Типы баз данных:**
- **Реляционные**: PostgreSQL, MySQL, MariaDB
- **NoSQL**: MongoDB (документная), Cassandra (ширококолоночная)
- **Key-Value**: Redis, etcd, FoundationDB
- **NewSQL**: CockroachDB, YugabyteDB (распределенные SQL)

**Инфраструктурные аспекты:**
- Репликация (master-slave, multi-master)
- Шардирование (горизонтальное масштабирование)
- Резервное копирование и восстановление
- Мониторинг производительности
- Connection pooling (PgBouncer, ProxySQL)
