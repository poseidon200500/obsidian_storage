
# Как подружить In-memory storage и `unique`: история одной оптимизации

Есть сервис, работающий с огромным объемом данных в оперативной памяти. При старте приложение «поднимается» из тяжелого дампа, загружая сотни тысяч объектов в In-memory хранилище.

В какой-то момент масштаб трагедии стал очевиден: **наше хранилище распухло до нескольких гигабайт**, и детальный профилинг показал, что львиная доля этого объема — обычные строки (теги). Более того, около **70% этих строк — дубликаты**. Казалось бы, решение на поверхности — интернирование. Но у нас есть критическое условие: данные в системе крайне динамичны. Теги постоянно добавляются, изменяются и удаляются.

Обычная `map[string]string` для хранения уникальных копий здесь не подходит — она превратилась бы в «черную дыру», которая только копит строки, но никогда их не отдает, вызывая утечки памяти. Использовать `sync.Pool` для долгоживущих объектов тоже не выйдет. Нам был нужен механизм, который умеет хранить только одну копию строки, пока она используется, и **автоматически вычищать её**, как только на неё исчезает последняя ссылка.

С выходом Go 1.23 и появлением пакета `unique` показалось, что «серебряная пуля» найдена. Идея была проста: заменить прямое хранение строк на дескрипторы, которые под капотом управляют каноническими копиями данных.

**Было:**

Go

```
type Tag string
type TagStorage []Tag // Тысячи строк, многие из которых идентичны
```

**Стало:**

Go

```
type TagUnique unique.Handle[string]
type TagStorageUnique []TagUnique // Только указатели на уникальные значения
```

Первые тесты воодушевили — экономия памяти составила **75%** на малых объемах. Однако, как это часто бывает с «магическими» пакетами стандартной библиотеки, при масштабировании на реальные дампы мы столкнулись с деградацией производительности в 3 раза и странным поведением GC, которое едва не превратило нашу оптимизацию в тыкву.
